CLAIMSIGNAL INTELLIGENCE WIRING + AUDIO ERROR FIX

The current ClaimSignal app has core intelligence screens, but the intelligence is not wired correctly yet.

This is not just a cosmetic issue.

The Signal Engine, Claim Intelligence, Adjuster Intelligence, Carrier Intelligence, Playbook, Timeline, and Audio Upload workflow must actually connect to claim data, uploaded evidence, extracted fields, and rule-based or AI analysis.

Do not erase existing data.
Do not remove working features.
Do not rebuild the app from scratch.
Do not weaken login, Master access, RBAC, masking, audit logs, shared masked claim visibility, or security hardening.

FIX 1: CLAIMS PAGE MUST SHOW AI / SIGNAL DETECTION

When I open Claims, there is no visible AI detection.

The Claims page must not look like a plain claim table only.

Each claim row or claim card should show:

- AI / analysis status
- risk level
- risk score
- escalation readiness score
- missing documentation status
- timeline extraction status
- carrier friction indicator
- adjuster friction indicator
- vendor signal if available
- Signal Engine summary
- next recommended action

Possible statuses:

- AI analysis pending
- MVP rule-based analysis complete
- Timeline needs review
- Missing documents
- Ready for escalation review
- Low risk
- Needs more data

Add a clear action:

- Run AI Analysis
or
- Analyze Claim
or
- Run Signal Engine

This action should be available from:

- Claims list
- Claim detail
- Evidence upload result
- Signal Engine page

If LLM extraction is not fully connected, use MVP rule-based analysis from available claim fields and uploaded parsed text.

Label it honestly:

“MVP rule-based analysis”

Do not call rule-based analysis full AI if it is not using an LLM.

The Signal Engine currently shows no real working intelligence.

Signal Engine must show:

- denied claims
- claims missing documentation
- claims with timeline extraction needs
- claims with supplement opportunity
- claims with carrier friction

- risk level

If the system cannot calculate a value because data is missing, show:

Do not show meaningless 0 values unless they are truly calculated from real supporting data.

- Denial Rate: 0.0%

Fix behavior metric display rules:

- If manually entered, show “Manual intelligence”

Do not show 0.0 unless the system has enough data to support that calculation.

FIX 8: AUDIO UPLOAD RUNTIME ERROR

The Audio Upload or Evidence Upload workflow is throwing this runtime error:

“A <Select.Item /> must have a value prop that is not an empty string.”

Fix all Select components causing this issue.

Search for Select.Item components with:

value=""

or empty string values.

Replace empty string values with a non-empty safe value such as:

- none
- unassigned
- auto_match
- unknown
- pending
- other

Example:

Do not use:

<Select.Item value="">None</Select.Item>

Use:

<Select.Item value="none">None</Select.Item>

Then convert “none” to null or undefined in the submit handler if needed.

This must be fixed anywhere it appears:

- audio upload
- evidence upload
- claim pre-select
- document category
- claim assignment
- timeline event type
- Playbook fields
- Carrier / adjuster / vendor selectors
- any other Select component

After fixing, audio upload must not crash the app.
